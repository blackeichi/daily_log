# Daily Log

> 개인 생산성 관리를 위한 풀스택 웹 애플리케이션  
> **Next.js (App Router) · NestJS · Prisma · PostgreSQL**

Daily Log는 하루 회고, 할 일, 루틴, 로그, 칼로리 관리를 한 곳에서 다룰 수 있도록 만든 서비스입니다.  
단순 CRUD 구현에 그치지 않고, **로그 탐색 UX**, **인증 구조**, **상태 관리**, **대용량 리스트 렌더링**을 중심으로 설계하고 구현했습니다.

---

## 프로젝트 개요

- **프로젝트명**: Daily Log
- **형태**: 개인 사이드 프로젝트
- **구성**: Monorepo (Frontend / Backend)
- **목적**: 일상 기록, 루틴 관리, 할 일 추적, 로그 탐색을 통합한 생산성 서비스 구현

### 이런 점에 집중했습니다

- 로그 목록에서 **무한 스크롤**과 **가상화 테이블**을 적용한 데이터 탐색 UX
- **JWT + Refresh Token** 기반 인증 처리
- Next.js App Router 환경에서 **Server / Client 경계 설계**
- 재사용 가능한 UI와 상태 구조 구성
- 실제 서비스처럼 느껴지는 입력 흐름, 탐색 경험, 기록 관리 UX 구현

---

## 주요 기능

### 1) 회고 / 감정 기록

- 하루 단위 회고 작성
- 감정 상태 기록
- 누적된 일상 로그를 기반으로 개인 기록 관리

### 2) 할 일 관리

- 오늘 / 주간 / 월간 / 연간 단위 할 일 관리
- Drag & Drop 기반 정렬
- 일정 단위로 태스크를 나눠 관리할 수 있는 UI 제공

### 3) 루틴 관리

- 반복 수행할 루틴 템플릿 생성
- 루틴 복사 및 재사용
- 일상 루틴을 구조적으로 기록할 수 있도록 설계

### 4) 로그 탐색

- 날짜 범위, 제목 기반 검색
- 무한 스크롤을 통한 연속 탐색
- 가상화 테이블 기반 렌더링 최적화
- 엑셀 다운로드 기능 제공

### 5) 칼로리 관리

- 식단 및 칼로리 기록
- 월간 단위 데이터 확인
- 기록 기반 통계 확인

### 6) 사용자 설정

- 프로필 정보 확인 및 수정
- 비밀번호 변경
- 인증 기반 보호 페이지 접근 제어

## 구현 포인트

### 1. 대량 로그 조회 UX

로그 데이터는 누적될수록 목록 길이가 길어질 수 있기 때문에, 단순 리스트 렌더링보다 **탐색 경험과 렌더링 비용**을 함께 고려했습니다.

- 검색 조건(기간, 제목) 기반 필터링
- 무한 스크롤 적용
- 테이블 렌더링 비용을 줄이기 위한 가상화 처리
- 엑셀 다운로드 기능 제공

이 과정을 통해 단순히 “데이터를 보여주는 화면”이 아니라,  
**사용자가 많은 기록을 빠르게 찾고 활용할 수 있는 화면**을 목표로 구현했습니다.

### 2. 인증 구조

사용자 인증은 JWT 기반으로 구성했습니다.

- Access Token / Refresh Token 분리
- 로그인 이후 보호된 페이지 접근 제어
- 사용자 정보 및 설정 변경 API 분리
- 인증이 필요한 요청과 공개 요청의 책임 분리

### 3. 상태와 UI 구조

화면별 책임을 나누고 공통 UI를 재사용 가능한 형태로 구성하려고 했습니다.

- Jotai 기반 상태 관리
- 공통 컴포넌트 분리
- 입력 UI, 모달, 토스트 재사용
- Drag & Drop 기반 목록 인터랙션 구현

### 4. App Router 기반 설계

Next.js App Router를 사용하면서 Server Component와 Client Component의 역할을 분리해 보려고 했습니다.

- 데이터 중심 영역과 인터랙션 중심 영역을 구분
- 필요한 구간만 Client Component로 분리
- 화면 책임을 작게 나누어 유지보수성을 높이도록 구성

## 트러블슈팅 / 설계 고민

### 1. 로그 목록에서 가상화를 적용한 이유

로그 데이터가 많아질수록 전체 row를 한 번에 렌더링하면 성능 저하가 발생할 수 있기 때문에,  
가시 영역만 렌더링하도록 구성해 렌더링 비용을 줄이도록 설계했습니다.

### 2. 무한 스크롤과 탐색 UX를 함께 고려한 이유

기록형 서비스 특성상 사용자는 단순히 “다음 페이지 이동”보다,  
자연스럽게 과거 데이터를 이어서 탐색하는 경험이 더 중요하다고 판단했습니다.  
그래서 페이지네이션보다 무한 스크롤 UX를 우선 적용했습니다.

### 3. App Router 환경에서 Client 범위를 최소화한 이유

모든 화면을 Client Component로 구성하기보다,  
인터랙션이 필요한 영역만 Client로 두고 나머지는 역할에 맞게 분리해 초기 렌더링 부담과 구조 복잡도를 줄이고자 했습니다.

### 4. 상태 관리를 Jotai로 선택한 이유

복잡한 전역 상태보다 화면 중심의 작은 상태 조합이 더 적합하다고 판단해,  
가볍게 조합 가능한 Jotai를 사용했습니다.

## 실행 방법

### 1) 패키지 설치

```bash
pnpm install
```

### 2) 환경 변수 설정

각 앱에 필요한 환경 변수를 설정합니다.

예시:

```bash
# apps/backend/.env
DATABASE_URL=
JWT_SECRET=
JWT_REFRESH_SECRET=

# apps/frontend/.env.local
NEXT_PUBLIC_API_URL=
```

### 3) 개발 서버 실행

```bash
pnpm dev:backend
pnpm dev:frontend
```
